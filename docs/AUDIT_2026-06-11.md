# System Audit — 2026-06-11 (overnight)

**Scope:** full read-only audit of the haptic/audio research app ahead of (a) WashU IRB submission and (b) scaling to 100s of participants / 1000s of trials.
**Method:** code review of `server.py`, `db_handler.py`, `schema_v2.sql`, the JS frontend, plus a live browser walkthrough of all three pages against the running dev server. **No database writes were made and no app code was changed** — only docs were edited.

Severity scale: **Critical** (will lose/expose data or block the study), **High** (will bite soon, fix before IRB/scale), **Medium** (plan it), **Low** (hygiene).

## 1. Data integrity & durability

### 1.1 In-flight session data is memory-only — **High**

- `js/modules/testSession.js` keeps the whole session (trials, survey responses, EEG readings) in a JS object; it contains **zero** `localStorage` usage.
- Persistence happens only at session end/abort: `js/test/sessionPersistence.js:41-104` (`saveSessionData()` → localStorage + `POST /api/session`).
- There is no `beforeunload` handler anywhere in `js/test/` — an accidental refresh, navigation, or browser crash mid-session silently discards every completed trial up to that point, including ~30–60 min of participant time.
- Note the inverse case is handled OK: if the **server** dies, data survives in the browser and the save path falls back to localStorage with a warning toast (`sessionPersistence.js:92-98`).

**Fix:** checkpoint each completed trial to `localStorage` (or POST it incrementally) as it finishes; add a `beforeunload` confirm while a session is active.

### 1.2 Save transaction & failure handling — mostly sound

- `db_handler.save_session_data()` (`db_handler.py:943-1148`) performs the whole session in one transaction with `commit()` at the end and `rollback()` on any exception — a half-saved session cannot occur. Good.
- However a failed save has **no recovery path in the product**: the toast says "Data saved locally", but nothing in the frontend ever calls `POST /api/sessions/bulk` (grep: zero references in `js/`). Recovering localStorage sessions requires manual JSON export + hand-rolled curl. **Medium** — add a "sync local sessions" button or remove the dead bulk endpoint.
- Retrying a failed/duplicate save creates a **duplicate session**: the frontend session id is only embedded in the free-text `sessions.notes` field (`db_handler.py:1042`) and never checked for uniqueness. Double-saving is undetectable. **Medium** — store `frontend_session_id` as a real `UNIQUE` column and upsert.

### 1.3 Status/notes stored as parseable strings — **Medium (fragile)**

- Trial status and tester notes are concatenated into `trials.notes` — `"Status: completed | Tester notes: [...]"` (`db_handler.py:1074`) — then recovered by **regex** at read time (`_extract_trial_status` `db_handler.py:1162-1169`, tester-notes regex `db_handler.py:1503-1508`; frontend session id likewise `db_handler.py:1150-1159`).
- A tester note containing `|` or `]` or the literal string `Status:` corrupts parsing. These belong in real columns (`trials.status`, a `trial_tester_notes` table, `sessions.frontend_session_id`).

### 1.4 Backups — **High**

- `backups/` contains exactly one file: the pre-migration freeze `haptic_research_frozen_2026-06-08T183753Z.db` (created by `migrate_to_v2.py:63-67`). Nothing recurring.
- The live DB (`haptic_research_v2.db`) is gitignored (`.gitignore:6`), so git is not a backup either.
- Conversely `.gitignore:8-9` (`!backups/`, `!backups/**`) **force-includes DB snapshots in git** — see privacy finding 3.4.

**Fix:** nightly `sqlite3 haptic_research_v2.db ".backup backups/nightly-$(date).db"` via cron/launchd + copy to university-approved encrypted storage; keep snapshots out of git.

### 1.5 Schema v2 review (`schema_v2.sql`)

Good: FK declarations with `PRAGMA foreign_keys = ON` per connection, CHECK constraints on every survey enum and slider range, `UNIQUE (session_id, trial_order)`, sensible composite indexes on `brainwave_readings`.

Gaps:

| Issue | Evidence | Severity |
|---|---|---|
| Pattern tags/annotations key on **pattern name text**, not `pattern_id` (`pattern_analysis_tags.pattern_name`, `pattern_annotations.pattern_name`) — renaming an audio file silently orphans its classification history | `schema_v2.sql:115-126` | Medium |
| No index on `trials(pattern_id)` — used by the `/api/pattern-stats` join and any per-pattern query; table scan at 1000s of trials | `schema_v2.sql`, `db_handler.py:621-636` | Medium |
| No index on `sessions(session_date)` — `get_analysis_sessions` orders by it | `db_handler.py:1209` | Low |
| `brainwave_readings`/`trial_events` lack `ON DELETE CASCADE` (survey tables have it) — future trial deletion will orphan readings | `schema_v2.sql:128-156` | Low |
| `patterns.name` not unique (only `file_path` is); two paths with the same display name merge silently in all name-keyed analysis | `schema_v2.sql:22-29` | Low |
| `locations` accepts client-supplied `location_id` and auto-creates `"Location N"` rows | `db_handler.py:810-827` | Low |
| `ensure_participant` updates demographics with `if age or gender or ...` — falsy values (age `0`) never update, and any later session silently **overwrites** demographics for an existing code | `db_handler.py:779-800` | Low |
| `journal_mode` is `delete` (not WAL) and no `busy_timeout` — fine while single-threaded, must change before any concurrency (see 2.2) | measured via `PRAGMA journal_mode` | Medium |
| `ensure_analysis_schema()` runs on **every** `get_connection()`: PRAGMA table_info, 3× `CREATE TABLE IF NOT EXISTS`, a COUNT, and a legacy-migration probe per request | `db_handler.py:366-444` | Medium |

### 1.6 Exclusion / annotation flows — OK with caveats

- `set_trial_exclude_from_analysis` and `set_trial_analyst_notes` correctly 404 unknown trials and rollback (`db_handler.py:522-544`, `743-776`).
- No audit trail: exclusions have no `updated_at`/reason column; for IRB-grade data provenance you want *who/when/why* a trial was excluded (analyst note is free-text and optional). **Medium**.
- Two separate editors write the same analyst-note field (trial drawer in `trialsWorkbench.js:274-284` and `trialDetailView.js:100-107`) with no conflict detection — last write wins. **Low**.

## 2. Scale readiness

### 2.1 `/api/analysis/sessions` will not survive the planned scale — **Critical**

Measured tonight against the live server (19 sessions, 87 trials, 33,927 brainwave readings):

- **Payload: 12.7 MB JSON, 0.57 s server time** — ≈150 KB per trial, dominated by raw per-second band-power readings (each reading row serializes 14 floats × ~390 readings/trial).
- `get_analysis_sessions()` (`db_handler.py:1172-1525`) loads *every* session, *every* reading, *every* survey row into Python dicts, then `json.dumps` the whole thing in memory (`server.py:54`). No pagination is used by the client (the `?limit` param exists but `analysisDataLoader.js:23` never passes it).

Linear projection (readings dominate, growth is linear in trials):

| Trials | Payload | Implication |
|---|---|---|
| 87 (today) | 12.7 MB | 0.6 s — already sluggish on refresh |
| 500 | ~73 MB | 5–10 s server serialize + parse; multi-GB browser heap spikes |
| 2,000 | ~290 MB | browser tab likely OOMs; server blocked for tens of seconds |
| 5,000 | ~730 MB | unusable |

Practical breaking point: **a few hundred trials**, i.e. within the first weeks of scaled collection.

**Fix (in order of leverage):**
1. Split EEG readings out of the corpus payload — return sessions/trials/surveys only (~5–10 KB/trial), and fetch readings lazily per trial (`GET /api/analysis/trials/<id>/readings`) for the detail chart.
2. Server-side aggregation for the dashboard: per-pattern medians/quartiles/leanings (the exact shapes `analysisDataProcessor.js` computes) are cheap SQL `GROUP BY`s.
3. Downsample or pre-aggregate readings at write time (per-phase means already cover most charts).
4. gzip responses and add pagination as a backstop.
5. Note: `get_analysis_sessions` builds `IN (?,?,…)` lists with one placeholder per trial id (`db_handler.py:1263`) — SQLite's variable limit (999 in older builds, 32k in newer) becomes a hard runtime error at scale even if memory holds.

### 2.2 Single-threaded `http.server` — **High**

- `server.py:312` uses `socketserver.TCPServer` (not even `ThreadingHTTPServer`): **one request at a time**. While the analysis payload serializes (or a big audio file streams), every other request — including the Test page's API calls during a live participant session — queues behind it.
- This also means an Analyze user can freeze a concurrently running Test session's saves. With multiple researchers (the scaling plan), this is structural.

**Fix:** minimum one-liner — `class ThreadingTCPServer(socketserver.ThreadingMixIn, socketserver.TCPServer)` + SQLite WAL + `busy_timeout`. Better: move the API to Flask/FastAPI behind uvicorn; keep static file serving as-is.

### 2.3 Client-side aggregation cost — **Medium**

- `js/modules/analysisDataProcessor.js` (1,393 lines) re-derives flat trial lists per interaction; `getTrialDetail()` does a full corpus scan per click (`analysisDataProcessor.js:709-712`), and every filter change re-walks all sessions/readings. Fine at 87 trials; quadratic-feeling at thousands. Memoize `_getFlatTrials` per filter state, or (better) let the server aggregate (2.1.2).
- `js/dataCache.js` TTL is 60 s — the 12.7 MB corpus is re-downloaded on nearly every Analyze visit.

### 2.4 Audio catalog inconsistency — **Medium**

- Active catalog: `audio_files/` = **21 files** (AUD001–AUD021). Legacy: `legacy_audio_files/` = 429 files / 89 MB.
- `audio-files.json` (the documented static-hosting fallback) still lists **425 legacy entries** with paths into `audio_files/` that no longer exist — any consumer of the fallback gets a 100% broken catalog.
- `pattern_metadata.json` holds 446 entries (425 stale + 21 active); `get_pattern_metadata_catalog()` (`db_handler.py:496-519`) ships all of them to Analyze on every load.
- DB `patterns` table has 20 rows keyed by historical `file_path` — pattern identity across catalog migrations rests entirely on file naming discipline (see 1.5 name-keying).

**Fix:** regenerate `audio-files.json` and prune `pattern_metadata.json` to the active catalog (script exists: `generate_metadata.py`); decide the canonical pattern identity (id, not name/path) before scaling the catalog.

## 3. Security & privacy (IRB lens)

### 3.1 Open network surface — **Critical**

- `server.py:312`: `socketserver.TCPServer(("", PORT), …)` binds **0.0.0.0** — the API is reachable by every device on the same Wi-Fi/LAN.
- There is **no authentication on any endpoint** (`server.py` has no auth code at all), and `Access-Control-Allow-Origin: *` (`server.py:50`) lets any website opened in a lab browser read the API cross-origin.
- Consequences for IRB: anyone on the network can dump `/api/participants` (codes, age, gender, handedness, notes) and `/api/analysis/sessions` (full study data incl. demographics per session), and can **write** (create sessions, tags, overwrite analyst notes, toggle exclusions) — i.e., research-record integrity is unprotectable as deployed.
- `eeg_server.py:446` correctly binds `localhost` — only the HTTP server is exposed.

**Fix (lab-tool appropriate):**
1. Bind `127.0.0.1` (one-line change). If multi-machine access is needed, front it with an SSH tunnel or tailscale.
2. Drop the CORS wildcard (same-origin pages don't need CORS at all).
3. If it must sit on a LAN: add a shared-secret header check on every request and a write-token for POSTs. Document this in the IRB data-security plan.

### 3.2 PII inventory & exposure — **High**

What exists (all in `haptic_research_v2.db`):

| Data | Where | Exposed via |
|---|---|---|
| Participant code, age, gender, handedness, free-text notes | `participants` (`schema_v2.sql:5-13`) | `/api/participants`, embedded in every `/api/analysis/sessions` session (`db_handler.py:1428-1432`) |
| Experimenter name, session notes | `sessions` | `/api/analysis/sessions` |
| Location name/address | `locations` (`schema_v2.sql:15-20`) | `/api/locations` |
| Full session copies incl. demographics | `localStorage['sessions']` in the lab browser profile | anyone at the machine |

Recommendations:
- Keep the code↔identity key **outside this system** (paper or separate encrypted store); state that in the protocol.
- Free-text fields (participant notes, session notes, tester notes) are the realistic re-identification vector — add UI guidance ("no identifying information") and an IRB-documented review step.
- Demographics don't need to ride along in the analysis payload; return them only where needed.
- Purge `localStorage['sessions']` after confirmed DB save, or encrypt at rest if it must stay.
- Plan a retention/deletion story (no delete path exists in app or API today).

### 3.3 Stored XSS — participant text executes in the analyst's browser — **High**

The server does **not** sanitize HTML (e.g. `title_case_action`, `db_handler.py:217-226`, only re-cases text: `<img src=x onerror=…>` survives as `<img Src=x Onerror=…>`). Escaping is therefore the renderer's job, and several Analyze components skip it:

| Sink | Data origin | Evidence |
|---|---|---|
| Trial detail "Other actions" `${actions}` | **participant-typed** custom survey actions | `trialDetailView.js:27,78` |
| Trial detail context fields via `_text()` (no escaping): participant notes, session notes, experimenter, location, equipment | researcher-typed | `trialDetailView.js:67-72,90,247-250` |
| Tester notes `${this._formatTesterNotes(...)}` | researcher-typed mid-session notes | `trialDetailView.js:91,252-258` |
| Participant chip `${participant.code}` | researcher-typed code | `analyzeToolbar.js:222-232` |
| Pattern name in the execution overlay | filenames (researcher-controlled) | `testExecutionOverlay.js:586,637` |
| `style="--tag-color:${tag.color}"` — color accepted verbatim from `POST /api/analysis/tags` (`db_handler.py:572-607` doesn't validate) | API client | `patternTagsPanel.js:101` |

By contrast `trialsWorkbench.js`, `patternTagsPanel.js`, `wordCloudPanel.js`, and `uiHelpers.js` escape correctly — the codebase has **five duplicated `_escape` implementations** with differing completeness (`trialDetailView.js:222-227` omits quote escaping) instead of one shared helper.

**Fix:** export one `escapeHtml` (it already exists in `uiHelpers.js:24-30`), use it at every interpolation of stored text; validate `color` as `#rrggbb` server-side. Even for a trusted lab, this protects analysts from accidental markup in notes corrupting the UI.

### 3.4 Snapshots with participant data destined for git — **High (IRB)**

- `.gitignore:8-9` force-includes `backups/**`; the frozen DB (full participant table) currently shows as untracked and **will be committed by the next `git add .`**. `docs/CONTRIBUTING.md` describes a GitHub-hosted workflow — pushing would move participant data to third-party hosting, which IRB data plans typically prohibit.
- **Fix:** ignore `backups/*.db`; verify no DB blob is already in history before the repo is ever pushed.

### 3.5 Lesser items

- Error responses return full Python tracebacks to the client (`server.py:199-204`, and `save_session_data` appends `traceback.format_exc()` to its error payload, `db_handler.py:1140-1144`). **Low** — log server-side, return generic messages.
- `do_POST` reads the entire body into memory with no size cap (`server.py:178-179`) — a junk 2 GB POST takes the (single-threaded) server down. **Low/Medium**.
- No HTTPS — acceptable for 127.0.0.1; not acceptable if the bind stays `0.0.0.0`.
- `eeg_server.py` accepts any localhost WebSocket origin — fine for a lab; note it.

A scripted browser walkthrough of the full setup → calibration → trial → survey flow was run against the live server (session aborted before any trial completed; nothing was written — DB row counts verified identical before/after). Findings below; icon-markup glitches were excluded per the concurrent icon work.

### 4.1 EEG-disconnected behavior

- With `eeg_server.py` not connected, the Test page surfaces stall/stale-data states in the control panel (`js/test/testExecution.js:293-325` defines "still running / no new EEG samples for Ns" warnings) — the overlay relies on the researcher noticing the banner; the participant-facing overlay keeps its timers running regardless. Verify the team is comfortable that **trials proceed with zero readings recorded** rather than hard-stopping: `_is_persistable_trial` (`db_handler.py:881-892`) only requires a completed survey, so EEG-empty trials persist silently. **Medium** — consider a "no EEG data" badge on save and in Analyze.

### 4.2 Survey flow integrity

- The 12-step survey state machine in `trialTagsSurvey.js` enforces required steps client-side and the DB CHECK constraints backstop it (good defense in depth).
- Keyboard/focus order, slider behavior, overflow, and theme contrast issues observed live are listed in Appendix A.

### 4.3 Abort semantics — **worth a protocol note**

- Abort mid-stimulation requires a second confirm (`testExecution.js:885-905`) — good. But abort **saves completed trials** ("Completed trials with participant feedback will be saved", `testExecution.js:876`) — researchers should know an aborted session still writes data and shows up in Analyze flagged `[ABORTED]` via a notes substring (`db_handler.py:1043-1044`, `1436`) rather than a real column (see 1.3).

## 5. Researcher experience

- **Server-down on Test page:** session save falls back to localStorage with a clear warning toast (`sessionPersistence.js:92-98`) — data is *not* lost, but see 1.2: there is no in-product way to sync it later.
- **Server-down on Analyze:** clean error banner with Retry (`analysisDataLoader.js:28-40`); stale cached corpus is shown if available — good. The 60 s cache means a just-saved session may not appear until refresh is clicked (no hint tells the researcher this).
- **Tag/note save failures** show inline "Save failed — try again" (`patternTagsPanel.js:215`, `trialDetailView.js:215`) without queuing or auto-retry; typed notes stay in the textarea (not lost) unless the researcher navigates away.
- **Duplicate participant flow** is validated client-side only (`sessionInfo.js:1398-1409` checks the cached list); two simultaneous Test machines could still race — acceptable now, races matter at scale.
- **Inconsistency:** the analyst note is editable in two places with different row heights and save buttons (workbench drawer vs. detail panel) — unify (1.6).
- Live UX observations (load time, navigation, charts) are in Appendix A.

## 7. Do this before IRB submission

1. **Bind the server to 127.0.0.1 and drop CORS `*`** (3.1) — one-line each; removes the network exposure of PII and unauthenticated writes.
2. **Fix `.gitignore` so `backups/*.db` never reaches git/GitHub** (3.4); verify history is clean.
3. **Write the data plan**: where the code↔identity key lives (outside this DB), free-text "no identifying info" rule, localStorage purge policy, retention/deletion (3.2).
4. **Automated nightly DB backup** to approved encrypted storage (1.4).
5. **Escape all stored text at render** — one shared `escapeHtml`, used in `trialDetailView` and `analyzeToolbar` first (3.3).
6. **Session checkpointing + `beforeunload` guard** on the Test page so a mid-session crash can't destroy a participant's hour (1.1).

## 8. Do this before scaling (100s of participants / 1000s of trials)

1. **Restructure `/api/analysis/sessions`**: strip raw readings from the corpus payload; add per-trial readings endpoint + server-side per-pattern aggregates (2.1). This is the single biggest engineering item.
2. **Threaded server + SQLite WAL + busy_timeout**, or migrate API to Flask/FastAPI (2.2).
3. **Schema hardening**: `frontend_session_id UNIQUE` (dedupe saves), real `trials.status` column, tester-notes table, `trials(pattern_id)` index, key tags/annotations by `pattern_id` (1.3, 1.5).
4. **Move `ensure_analysis_schema` to startup** (1.5) and add the missing indexes.
5. **Reconcile the audio catalog** — regenerate `audio-files.json`, prune `pattern_metadata.json`, settle pattern identity (2.4).
6. **Decide the multi-researcher story**: concurrent Test machines + Analyze sessions implies auth, write tokens, and exclusion audit trails (1.6, 3.1).
7. Wire up or delete `/api/sessions/bulk`; add a local-session resync UI (1.2).
